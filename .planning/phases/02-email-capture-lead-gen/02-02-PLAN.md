Create the email subscribe endpoint that accepts form submission, validates input, generates verification token, sends confirmation email via Resend, and stores token:

1.  Endpoint setup:
    - File: src/app/api/email/subscribe/route.ts
    - Method: POST (reject GET, HEAD, etc with 405 Method Not Allowed)
    - Content-Type: application/json
    - Returns: JSON {status, message, ...} with appropriate HTTP status

2.  Function body (async POST handler):

    a. Parse request body: const body = await request.json()

    b. Validate input using SubscribeRequest schema:

    ```typescript
    const validationResult = SubscribeRequest.safeParse(body);
    if (!validationResult.success) {
      return Response.json(
        { status: "error", message: "Invalid email or missing consent" },
        { status: 400 },
      );
    }
    const { email, name, consent } = validationResult.data;
    ```

    c. Generate opaque verification token:
    - Use: `crypto.randomBytes(32).toString('hex')` (generates random 64-char hex string)
    - Token must be unique and non-guessable

    d. Store token in memory (simple Map):

    ```typescript
    const verificationTokens = new Map<
      string,
      { email: string; createdAt: Date; used: boolean }
    >();
    verificationTokens.set(token, {
      email,
      createdAt: new Date(),
      used: false,
    });
    ```

    - Note: In production, this would be a database (Prisma) or Redis. For v1, in-memory is acceptable (per RESEARCH.md).
    - Add expiry: tokens expire after 24 hours (check createdAt + 24h > now before accepting)

    e. Send confirmation email via Resend:

    ```typescript
    const resend = new Resend(process.env.RESEND_API_KEY);
    const verifyUrl = `${process.env.NEXT_PUBLIC_BASE_URL || "http://localhost:3000"}/?verified=true&token=${token}`;

    const sendResult = await resend.emails.send({
      from: "noreply@<your-domain>", // Use Resend sender domain (per Phase 2 research notes)
      to: email,
      subject: "Verify your email - Malt Keyword Tool",
      html: `<h2>Verify your email</h2><p>Click the link below to unlock full results:</p><a href="${verifyUrl}">Verify Email</a>`,
    });
    ```

    - If send fails, return 500 error with message "Failed to send confirmation email"
    - Use try/catch to handle Resend API errors

    f. Return success response:

    ```typescript
    return Response.json(
      {
        status: "success",
        message: "Confirmation email sent",
        verificationEmailSent: true,
      },
      { status: 200 },
    );
    ```

3.  Rate limiting (Claude's discretion — implement simple limit):
    - Track by email: max 3 subscribe requests per email per hour
    - If exceeded, return 429 with message "Too many requests. Please try again later."
    - Use in-memory rate limit store (or env var to disable in tests)

4.  Error handling:
    - Invalid JSON: 400 Bad Request
    - Missing/invalid email: 400 Bad Request
    - Consent not checked: 400 Bad Request
    - Resend API failure: 500 Internal Server Error (don't expose Resend details to client)
    - Rate limit exceeded: 429 Too Many Requests

5.  Security:
    - Validate RESEND_API_KEY exists (throw error if missing in production)
    - Don't log sensitive data (email, token) to stdout
    - Use HTTPS in production (per Resend requirement)

6.  Response format matches SubscribeResponse schema from Task 1

Create the email verify endpoint that validates verification tokens and confirms email ownership:

1.  Endpoint setup:
    - File: src/app/api/email/verify/route.ts
    - Method: GET (user clicks link in email: /?verified=true&token=ABC123)
    - Query parameters: token (required)
    - Returns: JSON {status, message, redirectUrl} with appropriate HTTP status

2.  Function body (async GET handler):

    a. Extract token from URL query:

    ```typescript
    const { searchParams } = new URL(request.url);
    const token = searchParams.get("token");
    ```

    b. Validate token exists:

    ```typescript
    if (!token) {
      return Response.json(
        { status: "error", message: "No verification token provided" },
        { status: 400 },
      );
    }
    ```

    c. Look up token in storage (from subscribe endpoint):

    ```typescript
    const tokenRecord = verificationTokens.get(token);
    ```

    d. Validate token:
    - Token exists in store
    - Token has not expired (created < 24 hours ago)
    - Token has not been used yet (used === false)
    - If any check fails, return 400 or 410 (Gone) with error message

    e. Mark token as used (to prevent replay attacks):

    ```typescript
    tokenRecord.used = true;
    ```

    f. Return success response with redirect:

    ```typescript
    return Response.json(
      {
        status: "success",
        message: "Email verified successfully",
        redirectUrl: "/?verified=true",
      },
      { status: 200 },
    );
    ```

3.  Client-side handling (NOT in this endpoint):
    - Page.tsx from Plan 01 will read ?verified=true from URL and call clearGate()
    - Optional: return 302 redirect from here instead of JSON, but JSON is simpler for testing

4.  Error handling:
    - No token provided: 400 Bad Request
    - Token not found: 400 or 404 Not Found (don't expose whether token exists)
    - Token expired: 410 Gone
    - Token already used: 410 Gone (prevent re-verification)
    - Other errors: 500 Internal Server Error

5.  Security:
    - Don't expose token details in error messages (e.g., "Token expired" is OK, "Token ABC123 not found" is bad)
    - Mark tokens as used immediately to prevent replay attacks
    - Log verification events (optional: for analytics)

6.  Response format matches VerifyResponse schema from Task 1
